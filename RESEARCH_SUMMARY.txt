================================================================================
KEY ARCHITECTURAL DECISIONS
================================================================================

1. GAME LOOP: FIXED TIMESTEP (SlashSaber Pattern)

   Why: Deterministic physics, prevents jitter, feels right for racing games

   Implementation:
   - Cannon.World.step(fixedTimeStep=1/60, actualDelta, iterations=3)
   - Game logic runs AFTER physics step
   - Variable delta for game math (speed acceleration)

   Code:
   ```
   const fixedTimeStep = 1.0 / 60.0;

   update(delta) {
       this.world.step(fixedTimeStep, delta, 3);

       for(const handler of this.logicHandlers) {
           handler(delta);
       }

       this.score += this.speed * delta;
   }
   ```

2. PHYSICS LIBRARY: CANNON-ES (Recommended)

   Instead of Cannon.js:
   - Modern ES modules
   - Better tree-shaking
   - Smaller bundle size
   - Same API
   - npm install cannon-es

   Only use Cannon.js if inheriting existing project

3. STATE MANAGEMENT: SINGLETON WITH EVENTS

   Why: Single source of truth, decoupled systems

   GameState Contains:
   - Physics world (CANNON.World)
   - Scene (THREE.Scene)
   - Game metrics (speed, score, distance, lifes)
   - Logic handler registry
   - Event dispatcher

   Benefits:
   - Obstacles don't need GameState reference (use events)
   - UI updates via listeners
   - Easy to add analytics
   - Clean separation of concerns

4. OBSTACLE SYSTEM: FIXED POOL WITH STREAMING

   Why: Prevents garbage collection pauses, consistent performance

   Pool Strategy:
   - maxObstacles = 15 (fixed size)
   - Always spawn when count < max
   - Distance-based spawning (not time-based)
   - Automatic despawn when out of bounds

   Distance Calculation:
   - Normal mode: 2.5-3.0 units apart
   - Rush mode: 1.8-2.3 units apart
   - Randomized: Math.random() * (max - min) + min

   Placement Rotation:
   - LEFT, CENTER, RIGHT (prevent repetition)
   - Track last-used with cooldown
   - Select least-recently-used placement
   - Prevents 3+ obstacles in a row on same side

5. PHYSICS-RENDERING SYNC: MANUAL

   Why: Allows game-specific transforms (obstacles moving forward)

   Pattern:
   ```
   for(const piece of slicedPieces) {
       piece.body.position.z += speed * delta;

       piece.model.position.copy(piece.body.position);
       piece.model.quaternion.copy(piece.body.quaternion);
   }
   ```

   Benefits:
   - Explicit control
   - Can apply additional transforms
   - Easy to debug desynchronization
   - Perfect for streaming games

6. COLLIDER SELECTION: SIMPLE FIRST

   Recommendation:
   1. Sphere (fastest, good for obstacles)
   2. Box (second fastest, good for terrain)
   3. Capsule (character controllers)
   4. ConvexCollider (complex static shapes)
   5. Trimesh (ONLY for static terrain)

   NEVER use Trimesh for dynamic bodies (too slow)

================================================================================
PERFORMANCE OPTIMIZATIONS
================================================================================

Physics World Setup:
- world.broadphase = new CANNON.SAPBroadphase(world)  // Fast collision detection
- world.solver.iterations = 10  // Accuracy/performance tradeoff
- world.allowSleep = true  // Deactivate inactive bodies
- Fixed timestep = 1/60  // Consistent frame rate

Rendering Setup:
- powerPreference: 'high-performance'
- pixelRatio cap: Math.min(window.devicePixelRatio, 1.5)
- antialias: true
- SRGB color space
- Clear color: black (no transparency needed)

Memory Management:
- Object pooling (fixed obstacle count)
- Reuse model clones (don't create new geometries)
- Remove from scene BEFORE removing from physics
- Explicit cleanup in update loops
